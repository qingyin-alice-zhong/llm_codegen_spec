# Multi-Backend LLM Code Generation Framework

This framework now supports multiple hardware backends through a clean abstraction layer. You can generate optimized kernel code for different hardware targets using the same LLM interface.

## Architecture Overview

```
┌─────────────────┐    ┌──────────────────┐    ┌─────────────────┐
│   main.py       │    │  KernelCodeGen   │    │     LLM         │
│  (Orchestration)│────│   (Core Logic)   │────│   (AI Models)   │
└─────────────────┘    └──────────────────┘    └─────────────────┘
         │                        │
         │                        │
         ▼                        ▼
┌─────────────────┐    ┌──────────────────┐
│ BackendFactory  │    │    Backend       │

                                │

            │    MKL)      │ │    NPU)      │ │              │

```

## Available Backends

### CPU Backend (`cpu`)
- **Target**: Intel Xeon CPUs with MKL optimization
- **Language**: C 
- **Compilation**: GCC + Intel MKL
- **Kernels**: PolyBench linear algebra kernels
- **Optimizations**: Vectorization, MKL BLAS calls, cache optimization

### NPU Backend (`npu`)  
- **Target**: AMD Ryzen AI NPU
- **Language**: C++
- **Runtime**: NPU-optimized compilation and execution
- **Kernels**: AI Engine kernels for NPU
- **Optimizations**: Quantization, NPU-specific APIs, memory layout

## Usage Examples

### CPU Backend
```bash
# Single kernel for CPU
python main_refactored.py --provider openai --model gpt-4o --backend cpu --kernel gemm

# All CPU kernels
python main_refactored.py --provider openai --model gpt-4o --backend cpu --all

# Multiple specific kernels
python main_refactored.py --provider google --model gemini-2.5-flash-preview-05-20 --backend cpu --kernel gemm syrk atax
```

### NPU Backend
```bash
# Single kernel for NPU
python main_refactored.py --provider openai --model gpt-4o --backend npu --kernel scale

# All NPU kernels
python main_refactored.py --provider openai --model gpt-4o --backend npu --all
```

## Backend Interface

The new simplified backend interface requires only **two main methods** for each backend implementation:

```python
class Backend(ABC):
    @abstractmethod
    def get_available_kernels(self) -> Dict[str, str]:
        """Return available kernels for this backend"""
        
    @abstractmethod  
    def create_prompt(self, complete_code: str, kernel_name: str) -> str:
        """
        Create backend-specific optimization prompt for the LLM.
        
        Args:
            complete_code: The original complete source code
            kernel_name: Name of the kernel to optimize
            
        Returns:
            Formatted prompt string for the LLM
        """
        
    @abstractmethod
    def create_output(self, complete_code: str, optimized_code_generated: str, kernel_name: str) -> str:
        """
        Create the final optimized complete code.
        
        Args:
            complete_code: The original complete source code
            optimized_code_generated: The optimized code generated by the LLM
            kernel_name: Name of the kernel
            
        Returns:
            The complete optimized code ready for compilation/execution
        """
        
    @abstractmethod
    def compile_and_run(self, optimized_file_path: str, kernel_name: str, 
                       output_dir: str) -> Dict:
        """Compile and run optimized code"""
        
    @abstractmethod
    def analyze_run_output(self, run_output: str) -> Dict:
        """Parse performance results from execution output"""
```

### Key Simplifications:

1. **`create_prompt()`**: Takes the complete code and kernel name, internally handles function extraction, signature analysis, and creates the backend-specific prompt.

2. **`create_output()`**: Takes the complete code and LLM-generated optimized code, internally handles function replacement and returns the final optimized complete code.

3. **Internal Implementation**: All code parsing, function extraction, and replacement logic is hidden inside each backend as private helper methods.

This design allows each backend to handle code structure differently (C vs C++ vs Python vs CUDA) without exposing implementation details to the core framework.

## Benefits of This Architecture

### ✅ **Clean Abstraction**
- Only two main methods needed for each backend
- Implementation details hidden inside each backend
- Backend-specific code parsing and manipulation

### ✅ **Code Reuse**
- Shared LLM interface, logging, result management
- Common workflow orchestration
- Reusable prompt patterns

### ✅ **Flexibility**  
- Each backend can handle different languages (C/C++/Python/CUDA)
- Different function extraction and replacement strategies
- Backend-specific optimizations and compilation

### ✅ **Extensibility**  
- Add new backends without touching core logic
- Different languages/toolchains per backend
- Easy A/B testing across backends

### ✅ **Maintainability**
- Single codebase for all backends
- Consistent CLI interface
- Centralized configuration

## Migration Path

The original `main.py` and `agent.py` are preserved for backward compatibility. The new architecture is in `main_refactored.py` and `agent_refactored.py`.

You can gradually migrate by:
1. Testing new backends with `main_refactored.py`
2. Comparing results between old and new implementations
3. Eventually replacing the original files

This gives you the benefits of a clean architecture while maintaining compatibility with existing workflows.
